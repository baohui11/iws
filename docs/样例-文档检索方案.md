# 文档检索方案（样例 · 节选）

> **说明**：本文为**架构级节选样例**，用于说明「项目文件如何被索引与检索」；接口参数、环境变量与部署细节以专项《检索接口文档》及代码为准。

---

## 1. 目标与范围

**目标**：用户在上传或更新项目文件后，可在系统内对**文件名与正文内容**做全文检索（示例）。

**范围外（示例）**：跨租户检索、公网搜索引擎收录、实时协同编辑。

---

## 2. 总体架构（节选）

```text
业务应用 (Next.js)
      │
      │  HTTP（检索 API）
      ▼
检索服务 ──► Meilisearch（索引）
      ▲
      │  抽取正文 / 建索引
      ▼
索引 Worker（Python 等）
      ▲
      │  队列 / 任务表
      ▼
对象存储（Supabase Storage）+ 文件元数据（Postgres）
```

**要点（示例）**：  
- 文件二进制在 **Storage**；元数据与任务状态在 **Postgres**。  
- **Worker** 异步拉取任务，用 **Tika** 等抽取文本，再写入 **Meilisearch**。  
- 前端不直连 Meilisearch，统一走 **检索 API**（带鉴权）。

---

## 3. 数据流（节选）

1. 用户上传文件 → 写入 Storage + 业务表。  
2. 生成「建索引」任务（队列表或消息队列）→ `processing`。  
3. Worker 下载 → 解析 → 推送索引 → 更新 `success` / `failure`。  
4. 前端搜索 → 调用检索 API → 返回命中列表（含高亮、裁剪片段等，以接口定义为准）。

---

## 4. 检索能力（示例）

| 能力 | 说明 |
|------|------|
| 全文 | 对抽取后的正文建索引 |
| 过滤 | 按项目、bucket、文件类型等（字段以索引 schema 为准） |
| 排序 | 默认按相关性；可扩展时间 |

---

## 5. 非功能（节选）

- **一致性**：索引滞后于上传属预期；UI 可展示「处理中」状态。  
- **失败重试**：任务失败可记录原因并支持人工或自动重试（策略在详细设计中落地）。  
- **安全**：检索 API 校验用户与项目权限，避免越权搜到其他项目文件。

---

## 6. 与 Next.js 的衔接（示例）

- 服务端通过环境变量配置 **`SEARCH_API_BASE_URL`**、**`SEARCH_API_KEY`** 等，由 `lib/doc-search` 一类模块封装调用。  
- 不在浏览器暴露 Meilisearch 主密钥。

---
