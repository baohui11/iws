# 数据库表结构设计

## 1. 部门表

```sql
departments
  id            uuid PK DEFAULT gen_random_uuid()
  name          varchar(100) NOT NULL
  parent_id     uuid FK→departments  -- 支持多级部门，一级部门为 NULL
  sort_order    int DEFAULT 0        -- 同级排序
  created_at    timestamptz DEFAULT now()
  updated_at    timestamptz DEFAULT now()
```

---

## 2. 用户表

```sql
profiles
  id              uuid PK FK→auth.users  -- 与 Supabase Auth 的 user id 对应
  name            varchar(50) NOT NULL
  employee_id     varchar(50) UNIQUE      -- 工号
  phone           varchar(20)
  department_id   uuid FK→departments
  system_role     varchar(20) NOT NULL DEFAULT 'user'
                  -- 枚举值：super_admin / dept_admin / dept_ld / company_ld / user
  status          varchar(20) NOT NULL DEFAULT 'active'
                  -- 枚举值：active / inactive
  created_at      timestamptz DEFAULT now()
  updated_at      timestamptz DEFAULT now()
```

> `email` 不存在这里，直接从 `auth.users` 读取。
> 用户停用时只改 `status = inactive`，数据完整保留。

---

## 3. 项目表

```sql
projects
  id                uuid PK DEFAULT gen_random_uuid()
  project_no        varchar(50) UNIQUE NOT NULL
  name              varchar(200) NOT NULL
  department_id     uuid FK→departments
  description       text
  status            varchar(20) NOT NULL DEFAULT 'preparing'
                    -- 枚举值：preparing / active / completed / archived
  start_date        date
  expected_end_date date
  actual_end_date   date
  created_by        uuid FK→profiles
  created_at        timestamptz DEFAULT now()
  updated_at        timestamptz DEFAULT now()
```

---

## 4. 项目标签表

```sql
-- 标签字典，由管理员维护
project_tags
  id          uuid PK DEFAULT gen_random_uuid()
  name        varchar(50) NOT NULL UNIQUE
  created_at  timestamptz DEFAULT now()

-- 项目与标签的多对多关联
project_tag_relations
  project_id  uuid FK→projects
  tag_id      uuid FK→project_tags
  PRIMARY KEY (project_id, tag_id)
```

---

## 5. 项目成员表

```sql
project_members
  id            uuid PK DEFAULT gen_random_uuid()
  project_id    uuid FK→projects NOT NULL
  user_id       uuid FK→profiles NOT NULL
  project_role  varchar(20) NOT NULL
                -- 枚举值：pm / member / director / dept_ld
  joined_at     timestamptz DEFAULT now()
  UNIQUE (project_id, user_id)  -- 同一项目内一人只有一个角色
```

---

## 6. 合同成果文件清单表

```sql
contract_deliverables
  id          uuid PK DEFAULT gen_random_uuid()
  project_id  uuid FK→projects NOT NULL
  name        varchar(200) NOT NULL
  description text
  sort_order  int DEFAULT 0
  created_at  timestamptz DEFAULT now()
  updated_at  timestamptz DEFAULT now()
```

---

## 表关系总览

```
auth.users (Supabase)
    │ 1:1
    ▼
profiles ──── department_id ────► departments (自关联 parent_id)
    │
    │ 通过 project_members
    ▼
projects ──── department_id ────► departments
    │
    ├── project_tag_relations ──► project_tags
    │
    └── contract_deliverables
```

---

## 几个设计说明

**关于 `profiles` 和 `auth.users` 的关系**
- Supabase Auth 负责认证，`profiles` 存业务信息
- 建议在 Supabase 里设置触发器：新用户注册时自动在 `profiles` 插入一条基础记录
- `email` / `last_sign_in` 等认证信息直接查 `auth.users`，不做冗余

**关于部门层级**
- 用 `parent_id` 自关联支持多级，查询时用递归 CTE（Supabase 支持）
- 如果实际只用两级，这个结构也完全兼容

**关于枚举值**
- 这里用 `varchar` 而不是 Postgres 的 `ENUM` 类型，方便后续扩展角色值而不用改表结构
- 可以在应用层或加 `CHECK` 约束来保证值的合法性

**关于 `project_members` 的唯一约束**
- 一个用户在同一项目内只能有一个角色
- PM 的唯一性（每项目只有一个PM）建议在应用层校验，或加一个局部唯一索引：
```sql
CREATE UNIQUE INDEX one_pm_per_project 
ON project_members (project_id) 
WHERE project_role = 'pm';
```
