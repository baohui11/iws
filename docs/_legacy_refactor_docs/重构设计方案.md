# 周报文件系统 · 重构设计方案

> 版本：v1 · 状态：设计定稿（阶段 0）
> 适用范围：iws（Next.js 周报 / 文件管理系统）整体架构与代码结构重构

---

## 一、背景与目标

### 1.1 现状一句话

Next.js 16 (App Router) + React 19 + HeroUI + TypeScript；数据/认证/存储/队列全部依赖 **Supabase 全家桶**；另有一个独立 Python 服务 `doc-processor`（Tika 解析 + Meilisearch 索引 + 检索 HTTP 服务）。

### 1.2 本次重构目标

- **去运维负担 + 提升代码质量**，两者都要，接受**分阶段渐进式**推进。
- 让基础设施（数据库、认证、存储、检索、解析、队列）从「被业务代码硬编码引用」变为「可插拔的适配器」。
- 重建清晰的分层与模块边界，让「改一个功能」不再跨 5+ 个文件夹。

### 1.3 关键决策（已拍板）

| 议题 | 决策 |
|---|---|
| 核心目标 | 去运维负担 + 代码质量，**两者都要，分阶段慢慢来** |
| 数据迁移 | 有数据，但**可接受停机迁移 / 重置密码** |
| 全文检索 | nice-to-have；**Meilisearch 可保留也可换 PG**，走适配器，决策延后 |
| 数据访问层 | **Drizzle ORM** |
| 部署环境 | **单台自建服务器 / Docker Compose** |
| 文档解析 | 计划换**更轻量的解析服务**，走适配器替换 Tika |
| 应用层传输 | 当前用**薄 Server Action**；可能有外部调用但不确定 → **现在不改 API Route，只保证可低成本切换** |

## 二、现状诊断（问题根因）

> 问题不在 Supabase「重」，而在**缺少清晰分层（领域/应用/基础设施）与模块边界**，加上基础设施被业务代码到处直接 import。

1. **没有领域/服务层，业务逻辑堆在 Server Action 里**
   典型：`actions/weekly/report-editor.action.ts` 的 `upsertWeeklyReportItemAction` 单函数混合了鉴权 + 周报规则（半天槽校验、跨项目冲突、逾期判断、状态机）+ 直接 `supabase.from()` 写库。无法复用、无法单测。
2. **数据访问不统一**：部分 action 调 `lib/db` 仓储函数，部分直接 `supabase.from()` 裸查，两套风格混用。
3. **`requireProfile()` 在多个 action 文件被复制粘贴**（reports / report-editor… 一字不差），缺少统一的「当前用户上下文」。
4. **鉴权散落**：`isUserProjectMember`、`projectHasPm`、`report.user_id !== profile.id` 等判断在每个 action 内联手写，无集中策略。
5. **缺输入校验体系**（无 zod），靠手写 `.trim()` 和 if；类型为手写 interface，易与 DB schema 漂移。
6. **目录按技术分层切**（`actions/`、`components/`、`lib/db/`、`types/`、`constants/` 再各自按域分），单个功能分散在多个文件夹。

### 现状对 Supabase 的耦合点

| Supabase 能力 | 当前用法 | 替代方向 |
|---|---|---|
| PostgREST `.from()/.rpc()` | ~75 文件直接用 JS client 查询 | Drizzle ORM（repo 层） |
| Auth | 会话/SSR cookie、`auth.admin` 建号/邀请/改密、`auth.users` 外键 | Auth.js + bcrypt（可重置密码，免迁旧哈希） |
| RLS | 大量行级策略做鉴权 | 上移到应用层 `core/auth` 策略 |
| Storage | 头像桶、项目文件桶 | MinIO（S3 兼容）或本地盘 |
| 队列 pgmq + 触发器 | DB 触发器入队 → Python worker 消费 | pgmq 保留 或 轻量表队列 |
| Meilisearch | 中文分词全文检索 + 高亮 | 保留 Meili 或切 PG（pg_trgm / pg_jieba），走适配器 |
| Tika 解析 | 提取正文 | 换更轻量解析服务，走适配器 |

## 三、目标架构

### 3.1 核心思想

1. **Feature-first（按功能模块组织）**，而非按技术分层切。
2. **端口/适配器（Ports & Adapters）**：DB、存储、检索、解析、队列、Auth 全部走接口，业务层只依赖接口，实现可插拔。

### 3.2 分层（依赖只能自上而下）

```
表现层    app/ (页面/路由)  ·  components/ (纯 UI)
   │
应用层    server actions / route handlers
          —— 薄传输层：解析入参(zod) → 调 service → 返回统一 Result
   │
领域层    modules/<域>/service.ts
          —— 业务规则 + 鉴权策略；不认识 supabase / drizzle / next 运行时
   │
基础设施  adapters：db(Drizzle) · auth · storage · search · parser · queue
          —— 实现统一接口(port)，业务层只依赖接口
```

### 3.3 目录结构（feature-first）

```
src/
  modules/
    auth/          # 认证、当前用户、profile
    projects/      # 项目、成员、合同成果
    weekly/        # 周报、事项、审批、豁免、考勤
    files/         # 文件、版本、预览、互动、评论、下载
    stats/         # 统计、导出
    admin/         # 部门、用户、导入
      <域内统一约定>:
        action.ts / route.ts   # 薄传输层（按需）
        service.ts             # 业务 + 鉴权
        repo.ts                # 数据访问（Drizzle）
        schema.ts              # zod 入参校验
        types.ts               # 域内类型（尽量从 db schema 推导）
        components/            # 仅本域用的 UI
  core/            # 跨域基础设施 + 约定
    db/            # Drizzle client + schema（由 init.sql 生成）
    auth/          # getCurrentUser() / requireUser() / can() 权限策略
    storage/       # StoragePort + 实现（MinIO / 本地）
    search/        # SearchPort + 实现（Meili 适配器 ↔ PG 适配器）
    parsing/       # ParserPort + 实现（轻量解析服务适配器）
    queue/         # QueuePort + 实现（pgmq / 表队列）
    result.ts      # 统一返回类型（现有 handleAction 升级）
    errors.ts      # 错误类型（带稳定 code）
  app/             # 路由/页面（保持 Next 约定）
  components/       # 跨域共享 UI
```

### 3.4 关键约定

- **Action / Route 都是薄传输层**：只做 `zod 解析入参 → 调 service → 包 Result`。业务逻辑一律下沉 service。
- **Service 与框架解耦**：service 只接受「纯参数 + `ctx`（当前用户等上下文）」，**不碰 `cookies()` / `NextRequest` / `revalidatePath`** 等 Next 运行时 API。这些只允许出现在 action/route 薄壳里。
  - 收益：将来要对外开放 HTTP API，只需给同一 service 再挂一个 `route.ts`，action 与 route 可并存共用 service。
- **鉴权集中**：`core/auth` 提供 `requireUser()` 与 `can(user, action, resource)`，替掉到处复制的 `requireProfile` 和内联判断。
- **数据访问只在 repo 层**：service 不直接碰 Drizzle/SQL；**任何地方不再出现裸 `supabase.from()`**。
- **DB schema 为唯一事实来源**：类型从 Drizzle schema 推导，手写 `types/` 大幅收敛。
- **基础设施全部藏在 Port 后面**：Meili 是否保留、解析用哪个服务、存储/队列实现，都是配置/适配器选择，业务零改动。
- **Result/错误码 HTTP 友好**：`Result` 携带稳定 `code`（而不仅是中文 message），将来 route 可直接映射 HTTP 状态码，零返工。

## 四、应用层传输：Action vs Route 决策

- **现状**：纯薄 Server Action（Web 前端用），DX 好、类型安全、样板少。
- **外部调用**：可能有（移动端/系统集成），但不确定 → **现在不预建外部 API**，避免过度设计。
- **策略：留接口、不预建**
  1. 现在 Web 用 Server Action，但 service 必须框架解耦（见 3.4）。
  2. 将来真要对外，给同一 service 挂 `route.ts`：service 不变，鉴权来源从 session 换成 API Key/token，入参从函数参数换成 `req.json()`。
  3. 何时加 route：等真实需求出现，且只加被需要的端点。
  4. 唯一值得「现在为未来做」的小投入：把 `Result`/错误码设计成 HTTP 友好（稳定 `code`）。

## 五、基础设施端口（Port 接口草案）

> 仅为方向性签名，阶段 1 定稿。

```ts
// core/result.ts
export type Result<T> =
  | { success: true; data: T }
  | { success: false; code: string; message: string }

// core/auth — 当前用户上下文 + 权限策略
export interface AuthPort {
  getCurrentUser(): Promise<AuthUser | null>
  // 建号/邀请/改密/登出等（替代 supabase.auth.admin）
  createUser(input: CreateUserInput): Promise<AuthUser>
  setPassword(userId: string, password: string): Promise<void>
}
export function requireUser(): Promise<AuthUser>          // 无则抛 AuthError
export function can(user: AuthUser, action: string, resource: unknown): boolean

// core/storage
export interface StoragePort {
  upload(p: { bucket: string; key: string; body: Buffer; contentType: string; upsert?: boolean }): Promise<void>
  download(p: { bucket: string; key: string }): Promise<Buffer>
  getSignedUrl(p: { bucket: string; key: string; expiresInSec?: number }): Promise<string>
  remove(p: { bucket: string; key: string }): Promise<void>
}

// core/search
export interface SearchPort {
  index(doc: SearchDoc): Promise<void>
  remove(id: string): Promise<void>
  search(q: SearchQuery): Promise<SearchResult>   // Meili 实现 ↔ PG 实现
}

// core/parsing — 提取文档正文（替代 Tika）
export interface ParserPort {
  extractText(p: { buffer: Buffer; ext: string; fileName: string }): Promise<{ content: string; degraded: boolean }>
}

// core/queue
export interface QueuePort {
  enqueue(queue: string, payload: unknown): Promise<void>
  // worker 侧：拉取/确认
  pull(queue: string, max: number): Promise<QueueMessage[]>
  ack(queue: string, msgId: string): Promise<void>
}
```

## 六、分阶段路线图（架构优先）

> 原则：**先立架构骨架与分层，再在新骨架内逐域迁移**；基础设施替换降级为「实现某个 Port」。每个阶段尽量可独立上线。

| 阶段 | 名称 | 内容 | 产出/可上线 |
|---|---|---|---|
| 0 | 设计定稿 | 确认目标架构、模块边界、各 Port 接口签名 | 本文档 |
| 1 | 立骨架 | 建 `core/`（Drizzle client + 由 init.sql 生成 schema、`result`、`errors`、`requireUser`、各 Port 接口 + 现有 Supabase 实现包一层）。**行为不变，只把 Supabase 收进适配器** | 解开「到处硬编码」死结 |
| 2 | 逐域重构（核心） | 一次一个域（建议 `auth → projects → weekly → files → stats → admin`），把 action 拆为 `action(薄)+service(业务+鉴权)+repo(数据)`，加 zod 校验，组件搬进模块 | 每域可独立上线 |
| 3 | 换数据库实现 | repo 底层从 PostgREST 适配器切到 Drizzle/自建 PG；RLS 鉴权上移 `core/auth` | 去 PostgREST |
| 4 | 换 Auth 实现 | Auth.js 实现 `AuthPort`，替掉 Supabase Auth（可重置密码，免迁旧哈希） | 去 Supabase Auth |
| 5 | 换存储/队列实现 | `StoragePort`→MinIO/本地盘；`QueuePort`→pgmq 或表队列 | 去 Supabase Storage |
| 6 | 检索与解析 | `SearchPort` 双实现（保留 Meili 或切 PG，配置切换）；`ParserPort` 接入更轻量解析服务替掉 Tika | 解析瘦身 |
| 7 | 清理 | 移除 Supabase 依赖（若决定去掉）、统一规范、补测试与部署文档、一套 docker-compose 起全栈 | 收尾 |

**关键好处**：阶段 1 完成后，「换不换 Supabase / 留不留 Meili / 用哪个解析服务」不再阻塞重构，全部退化为可延后、可灰度的适配器决策。

## 八、风险与注意事项

- **业务复杂度**：周报/审批/角色规则不简单，**务必渐进式、逐域迁移**，避免一次性重写。
- **鉴权下移**：从 RLS 迁到应用层时，需逐条核对原 RLS 策略，避免越权。建议在阶段 2/3 为每个域补鉴权用例。
- **数据迁移**：可停机 + 可重置密码，使 Auth 迁移大幅简化（不迁旧密码哈希，改为统一重置/初始密码）。
- **检索中文分词**：若最终切 PG，`pg_trgm` 简单够用；要更好分词需 `pg_jieba`/`zhparser`（需镜像内编译扩展）。保留 Meili 则无此问题。
- **解析与预览**：换轻量解析服务时注意覆盖现有格式（doc/docx/ppt/pptx/xls/xlsx 等）与「仅索引文件名」的降级路径。
