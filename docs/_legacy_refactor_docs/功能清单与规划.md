# 周报文件系统 · 功能清单与规划

> 版本：v1 · 用途：梳理**现有功能**全貌，标注**计划调整项**，并纳入**拟新增功能**（周报系统 2.0 遗漏 + 2.X 更新）。
> 与《重构设计方案.md》配套：本文管「做什么（功能）」，那份管「怎么搭（架构）」。

---

## 一、角色与权限基础

**系统角色**（`system_roles`）：`user` 普通用户 / `dept_ld` 部门领导 / `dept_admin` 部门管理员 / `admin` 超级管理员
**项目角色**（`project_roles`）：`pm` 项目经理 / `member` 成员 / `director` 总监 / `sale_ld` 销售 LD

> 同一人在不同项目可有不同项目角色；周报按「项目 + 周次」分别填写。

## 三、拟新增功能（来自需求图）

> 分两批：**A. 2.0 遗漏功能**、**B. 2.X 更新功能**。下表含「与现有的关系」和「初步实现要点」，最终以评审为准。

### A. 2.0 遗漏功能

| # | 功能 | 需求描述 | 与现有关系 | 初步实现要点 |
|---|---|---|---|---|
| 1 | **成果水印** | 敏感文件在线预览时添加「中大咨询-人员姓名-日期」水印 | 扩展现有「在线预览」；已有 `is_confidential` 字段 | 预览渲染层叠加动态水印（前端覆盖层 + 必要时后端烧录）；按 `is_confidential` 触发；水印含当前查看者姓名 + 日期，防截留痕 |

### B. 2.X 更新功能

| # | 功能 | 需求描述 | 与现有关系 | 初步实现要点 |
|---|---|---|---|---|
| 2 | **项目基础信息维护（OA 同步）** | 按天定期从 OA 系统同步项目、人员、组别数据 | 替代现「手动维护 + CSV 导入」 | 定时同步任务（拉取 OA → upsert 项目/人员/组别）；需 OA 接口与字段映射；冲突/删除策略 |
| 3 | **人员权限管理（OA 导入 + 范围限制）** | PM 在 OA 调整项目人员名单，系统从 OA 全量导入项目人员；**仅对能源事业部内部人员开放填报权限** | 影响 `project_members` 与填报鉴权 | 成员名单以 OA 为准全量导入；填报权限增加「能源事业部内部人员」判定（部门归属 + 内/外部标记） |
| 4 | **人员分组标签（组别）** | 人员列表新增「组别」列，后台批量维护人员所属组别，便于按组别统计/筛选 | 现有 `departments` 之下的更细分组 | 新增「组别」维度（人员 ↔ 组别）；列表展示 + 批量维护 UI；统计/筛选接入组别 |
| 5 | **项目阶段筛选** | 周报支持按项目阶段（销售/实施）筛选 | **已有 `project_stage` 枚举（销售阶段/实施阶段）**，主要补 UI | 在「我的周报」「周报统计」加阶段筛选项；复用现有字段 |
| 6 | **项目合同信息维护** | 录入合同基础信息、合同周期，可手动调整合同时间节点，适配项目延期 | 现 `projects` 有 `contract_no`，`contract_deliverables` 有成果清单 | 扩展项目合同字段（合同周期/起止/节点）；项目详情「信息」Tab 增加维护入口 |
| 7 | **项目周期设置 + 节点提醒** | 设置项目时间节点，在中期检查/验收交付等重要节点提醒成员填报成果文件；未及时填报则**顶部强制弹窗**直至完成 | 复用通知体系；新增节点配置 | 项目时间节点配置 → 到点生成提醒/通知；未完成时登录后顶部强提醒弹窗（阻断式），直到上传对应成果 |
| 8 | **项目填报状态开关** | 项目「继续填报/停止填报」状态，可手动设置；统计页快速筛选已结束填报项目，减少无效筛查 | 与 `project_status` 概念不同（独立填报开关） | 项目新增「填报开关」字段；停止填报后该项目不再要求/统计周报；统计页加该筛选 |
| 9 | **文件分类标签** | 上传文件新增下拉分类标签：销售类、实施类、立项、可研、项目建设书等，规范归类 | **与现有 `file_source`/`is_deliverable` 有重叠**，需统一 | 明确「分类标签」与现有来源字段的关系（并存 or 取代）；上传表单加下拉；搜索/统计接入分类 |

## 四、功能调整建议汇总（现有功能的改动）

1. **认证体系**：登录/密码相关由 Supabase Auth 迁至 Auth.js（见重构方案阶段 4）。
2. **用户/项目来源**：由「手动 + CSV 导入」转向「OA 每日同步」（新增 #2、#3）。管理后台的增删改可能退为「只读 + 例外维护」。
3. **文件分类口径统一**：现有 `file_source`（client/internal/public/original）+ `is_deliverable` 与新增「文件分类标签」（#9）需要统一为一套清晰口径，避免双轨混乱。**建议优先决策**。
4. **周报筛选增强**：加入「项目阶段」（#5）「填报状态」（#8）两个筛选维度，贯穿「我的周报」与「统计」。
5. **预览安全**：敏感文件预览加水印（#1），与 `is_confidential` 联动。
6. **检索/解析**：检索后端与解析服务走适配器，便于保留 Meili 或换 PG、换轻量解析（见重构方案阶段 6）。

## 五、新增功能与重构阶段的建议映射

> 原则：**先重构地基，再在新骨架上做功能**，避免在旧结构上继续堆代码。

| 新增功能 | 依赖/建议时机 |
|---|---|
| #5 项目阶段筛选 | 轻量，可早做（字段已有），适合作为重构后某域的「首个新功能」验证骨架 |
| #9 文件分类标签 | 需先统一文件分类口径；建议随 `files` 域重构一并落地 |
| #1 成果水印 | 随 `files`/预览域；与存储/预览适配器协调 |
| #6 合同信息 / #8 填报开关 | 随 `projects` 域重构落地（加字段 + 信息 Tab） |
| #4 人员组别标签 | 随 `admin`/人员域；注意与 #2 OA 同步的数据来源协调 |
| #2 OA 同步 / #3 权限范围 | 较重、依赖外部 OA 接口；建议地基（DB/鉴权）就绪后专项推进 |
| #7 周期节点提醒 | 依赖通知体系 + 项目周期（#6）；建议在 #6 之后 |

## 六、待确认问题（需产品/业务拍板）

1. **文件分类口径（#9）**：新「分类标签」与现有 `file_source` + `is_deliverable` 是**并存**还是**取代**？标签是单选还是多选？分类项是否需要后台可配置？
2. **OA 同步（#2/#3）**：OA 是否提供接口（还是导出文件）？同步哪些实体与字段？人员/项目「删除」如何处理？「组别」由 OA 提供还是系统内维护？
3. **填报权限范围（#3）**：「能源事业部内部人员」如何界定（部门树某节点下？内/外部员工标记？）？外部人员是否完全不可登录，还是可登录但不可填报？
4. **填报状态开关（#8）** 与 `project_status`（active/preparing/completed/archived/suspended）的关系：是独立开关，还是复用现有状态？
5. **水印（#1）**：是前端覆盖层（体验好、可被绕过）还是后端烧录进 PDF（更安全、成本高）？哪些文件算「敏感」（仅 `is_confidential`？还是按分类）？
6. **节点提醒弹窗（#7）**：「顶部强制弹窗直至完成」是否为**阻断式**（不处理无法使用系统）？提醒对象是项目全体成员还是特定角色？
