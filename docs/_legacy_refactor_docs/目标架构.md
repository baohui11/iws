# 周报文件系统 · 目标架构（重构后）

> 版本：v1 · 状态：方向定稿，含少量待定项
> 配套文档：《重构设计方案.md》（路线图/约定）、《功能清单与规划.md》（功能）。本文描述**重构完成后的目标技术架构**。

## 一、总览

一句话：**Next.js 应用（feature-first + 端口/适配器） + 一体化 Postgres（关系数据 + pgvector 向量 + BM25/全文 + pgmq 队列） + MinIO 对象存储 + 两个外部转换服务（Gotenberg 预览、MinerU 文档解析）**，对内走 tRPC、对外走版本化 REST，目标是支撑「周报业务 + 文件管理 + AI RAG 混合检索」。

## 二、技术选型决策

| 维度 | 选型 | 备注 / 理由 |
|---|---|---|
| 前端/全栈框架 | Next.js 16 (App Router) + React 19 | 保留 |
| UI | HeroUI + Tailwind | 保留 |
| 数据访问 | **Drizzle ORM** | schema 由现库生成，唯一类型源 |
| 关系数据库 | **PostgreSQL（自建）** | 一体化底座 |
| 向量检索 | **pgvector** | RAG 语义召回 |
| 关键词检索 | **BM25（pg_search）或 tsvector + pg_jieba** | 中文分词，混合检索的关键词分支 |
| 混合检索 | **向量 + 关键词，RRF 融合** | 一条 SQL 出结果，无需外部索引同步 |
| 全文检索引擎 | **下线 Meilisearch**（经 `SearchPort` 保留可切回） | RAG 场景 PG 一体化更简单 |
| 认证 | **Auth.js + bcrypt** | 替代 Supabase Auth；可重置密码免迁旧哈希 |
| 鉴权 | **应用层策略（core/auth）** | 替代 RLS |
| 对象存储 | **MinIO（S3 兼容）+ 可选 S3 加密网关** | `StoragePort` 对加密无感 |
| 队列 | **pgmq**（统一）；文档管线可选 Celery/Redis | `QueuePort` 封装 |
| 定时任务 | **pg_cron**（或小型调度容器） | OA 同步、节点提醒 |
| 文档预览 | **Gotenberg**（Office→PDF，现成容器） | 给人看；主应用 worker 调用，无需自写服务 |
| 文档解析(RAG) | **自建解析微服务**（改造 knowledge_doc_converter）→ Markdown | 给机器读；OCR 用**线上 API**，不自部署 MinerU |
| OCR/解析引擎 | **线上 API（外部）**，藏在解析服务的 `OcrProvider` 接口后 | 厂商可换；按量计费，需限流/幂等 |
| 服务拆分 | **1 个自建微服务（解析）+ Gotenberg 现成容器** | 预览与解析职责/资源画像不同，分开 |
| Embedding | 中文 embedding 模型（如 bge-m3，OpenAI 兼容端点） | 经 `EmbeddingPort` |
| 对内传输 | **tRPC** | 类型安全 RPC |
| 对外传输 | **版本化 REST `/api/v1` + OpenAPI** | 第三方/移动端 |
| 部署 | **单机 Docker Compose** | profile 区分基础栈与 RAG 栈 |

## 三、分层与端口（与《重构设计方案》一致）

```
表现层    app/ · components/
应用层    tRPC router / REST route（薄）：解析入参(zod) → service → 统一 Result
领域层    modules/<域>/service：业务规则 + 鉴权；不依赖框架运行时与具体基础设施
基础设施  端口(Port) + 适配器(Adapter)：
          DbPort(Drizzle) · AuthPort · StoragePort · SearchPort ·
          ParserPort · PreviewPort · QueuePort · EmbeddingPort
```

> 核心不变量：**service 是业务逻辑唯一载体**；传输层（tRPC/REST）与基础设施（PG/MinIO/Gotenberg/MinerU/检索引擎）都可替换，业务零改动。

### 端口补充（相对设计方案新增）

```ts
// 预览：Office/各类格式 → 可在浏览器渲染的产物（PDF），含水印选项
export interface PreviewPort {
  toPdf(p: { buffer: Buffer; ext: string; watermark?: WatermarkSpec }): Promise<Buffer>
}

// 文档解析（RAG）：→ Markdown + 资产，便于切块
export interface ParserPort {
  toMarkdown(p: { buffer: Buffer; ext: string; fileName: string })
    : Promise<{ markdown: string; assets: AssetRef[]; degraded: boolean }>
}

// 向量化
export interface EmbeddingPort {
  embed(texts: string[]): Promise<number[][]>
}

// 检索：混合检索（向量 + 关键词 + RRF）
export interface SearchPort {
  upsertChunks(docId: string, chunks: Chunk[]): Promise<void>
  removeDoc(docId: string): Promise<void>
  hybridSearch(q: { query: string; filters?: SearchFilters; topK?: number }): Promise<SearchResult>
}
```

## 四、检索方案（RAG 混合检索）

### 存储模型（PG 内）

- `documents`：文档级元数据（关联现有 `files`）。
- `doc_chunks`：分块表，含 `content`(text)、`embedding`(vector)、`tsv`(tsvector，中文分词)、元数据（project_id/department_id/file_type 等用于过滤）。
- 索引：`embedding` 建 **HNSW**（pgvector）；`tsv` 建 **GIN**（或 pg_search 的 BM25 索引）。

### 检索流程

1. query → 同一 embedding 模型得到查询向量。
2. **向量召回**（pgvector，按业务过滤）取 topN。
3. **关键词召回**（BM25/tsvector，同样过滤）取 topN。
4. **RRF 融合**两路结果排序，取 topK。
5. （可选）**rerank** 精排后返回，供 RAG 上下文或检索结果页。

> 全部在一条/几条 SQL 内完成；无外部索引同步。`SearchPort` 封装，未来若要切 Meili/专用向量库仅替换实现。

### 写入流程（索引化）

文件上传 → 解析为 Markdown（ParserPort）→ 切块 → EmbeddingPort 批量向量化 → 写 `doc_chunks`（content + embedding + tsv）。全程异步走队列。

## 五、文档处理：两条管线、两类服务

预览（给人看）与解析（给机器读）性质完全不同，**拆开处理**。落地形态：**1 个自建微服务（解析）+ Gotenberg 现成容器（预览）**。

| 管线 | 目的 | 部署形态 | 职责边界 | 触发 |
|---|---|---|---|---|
| **预览管线** | 浏览器渲染 | **Gotenberg 现成容器** + 主应用 Node worker 薄编排 | Office→PDF；敏感文件烧录水印（#1）；回写 `preview_status` | 上传后异步 |
| **RAG 解析管线** | 检索/问答 | **自建解析微服务**（改造 `knowledge_doc_converter`） | 仅「**文件 → Markdown + 资源**」；内部调线上 OCR API | 上传后异步 |

### 为什么拆成两个

| 维度 | 解析（线上 OCR） | 预览（Gotenberg） |
|---|---|---|
| 性质 | I/O 等待型（等外部 API、轮询） | 本地 CPU 型 |
| 时长 | 长（分钟级） | 短（秒级，近交互） |
| 并发约束 | 受外部 API 限流/配额/计费 | 本地资源，可较大并发 |
| 失败模式 | 超时/配额/欠费 → 复杂重试 + 幂等 | 简单重跑 |
| 技术栈 | Python（复用 converter 生态） | 现成容器，零代码 |

合并的最大风险是**队头阻塞**：分钟级解析会堵住本应秒回的预览。拆开后可独立扩缩、独立限流、独立成本监控。

### 解析微服务设计要点（改造 knowledge_doc_converter）

- 复用其工程模式：**worker + HTTP 回调（不直连 DB） + S3 存产物 + 分布式锁 + 超时/重试 + Prometheus 指标**。
- 把原 `MinerUConfig` 那层抽象为 **`OcrProvider` 接口**：线上 OCR 厂商可随时切换，主应用无感。
- **职责单一到 Markdown 为止**：切块/向量化/写 `doc_chunks` 留在主应用（converter 原本也是「回调 markdown，后端再 dispatch 索引」的分法），保证 RAG 索引逻辑集中在数据旁边。
- 回调端点与 S3 改为对接 iws 后端 + MinIO。

## 六、队列与调度

- **主应用任务用 `pgmq`**（PG 内，`SKIP LOCKED`）：预览编排、切块+向量化、OA 同步、节点提醒、（重）索引重建。app 入队、Node worker 消费，`QueuePort` 封装。
- **解析微服务的队列**：沿用 converter 的 Celery（Redis broker）即可，**或**改造其 broker 直接消费 pgmq 以省掉 Redis。两种都经 `QueuePort`/回调对主应用透明。
  - 建议：若想基础设施最小化，优先把解析服务也接到 pgmq；若想最快复用 converter 现成代码，则接受 Redis 这一个额外依赖（仅服务于解析管线）。
- **并发与限流**：解析队列要按**线上 OCR API 的配额**限速 + 退避重试 + 幂等（同一文件不重复计费）。
- **定时**：`pg_cron`（或小型调度容器）驱动 OA 同步与节点提醒。

---

## 七、传输层（对内 tRPC / 对外 REST）

- **对内（Web 前端）**：tRPC router，薄层调用 service；端到端类型安全。
- **对外（第三方/移动端）**：`/api/v1/*` REST route handlers + OpenAPI 文档；鉴权用 API Key/Token。
- **统一约定**：两者都只做「zod 解析入参 → service → 统一 Result（带稳定 `code`，可映射 HTTP 状态码）」。
- 避免三套风格：Web 主用 tRPC，外部用 REST，Server Action 仅限极简表单（或不用）。

---

## 八、存储与加密

- **MinIO**（S3 兼容）为对象存储：源文件、预览 PDF、解析产物（Markdown/图片）、头像分桶。
- **加密**：MinIO SSE 或前置 **S3 加密网关**；app 只连一个 S3 endpoint，`StoragePort` 对加密无感。
- **访问**：预览/下载用**签名 URL**；直传也走同一 endpoint（经网关）。下载保留审计记录。

---

## 九、鉴权与安全

- 会话由 Auth.js 管理；`core/auth` 提供 `requireUser()` 与 `can(user, action, resource)` 策略，逐条对齐原 RLS 语义。
- 填报权限范围：支持「能源事业部内部人员」判定（待业务确认界定方式，见功能文档）。
- 敏感文件：`is_confidential` → 预览水印 + 下载审计。
- 对外 REST 用独立 API Key/Token，与会话隔离。

## 十一、关键数据流示例

**文件上传 → 可预览 + 可检索**
1. 用户上传 → 存 MinIO，写 `files` 记录。
2. 入队两类任务（pgmq）：预览生成、RAG 解析。
3. 预览 worker：调 Gotenberg 转 PDF（敏感文件加水印）→ 存 MinIO → 回写 `preview_status`。
4. 解析 worker：ParserPort 转 Markdown → 切块 → EmbeddingPort 向量化 → 写 `doc_chunks`。
5. 用户在「搜索文件」混合检索：SearchPort 向量+关键词+RRF → 返回高亮结果 / RAG 上下文。

## 十二、待定项（需拍板）

1. **关键词引擎**：`pg_search`(BM25) vs `tsvector + pg_jieba`（影响中文分词质量与扩展安装）。
2. **线上 OCR 厂商**：选哪家 OCR/解析 API（质量/价格/限流/数据合规），藏在 `OcrProvider` 后。
3. **解析服务队列**：接 pgmq（去 Redis，基础设施最小）vs 沿用 Celery/Redis（最快复用 converter）。
4. **切块+向量化放哪**：主应用 Node（推荐，边界干净）vs 解析微服务 Python（复用 llama_index）。
5. **水印实现**：PDF 烧录（安全）vs 前端覆盖层（便宜可绕过）；「敏感」判定口径。
6. **Embedding 模型**：本地自部署（bge-m3 等）vs 外部 API。
7. **Web 传输**：tRPC 全面替代 Server Action，还是二者共存过渡。
8. **OA 同步**：接口/字段映射、增量 vs 全量、删除策略（与功能文档联动）。

## 十三、模块划分（modules/）

按业务域 feature-first 组织，共 **7 个模块**；基础设施在 `core/ports` + `core/adapters`，不属于任何模块。

| 模块 | 职责范围 | 主要表 |
|---|---|---|
| `auth` ✅ | 登录/会话/密码、当前用户、个人资料、头像 | users（认证字段） |
| `org` | 部门、用户、（新）组别；组织架构与人员管理（含 admin 视角的增删改/导入） | departments, users |
| `projects` | 项目、成员、合同成果清单；（新）合同信息、周期节点、填报开关 | projects, project_members, contract_deliverables |
| `weekly` | 周报、工作/计划事项、审批、无工作周豁免、周次、考勤 | weekly_reports, weekly_report_items, weekly_report_approvals, project_week_exemptions, weeks |
| `files` | 文件、版本、预览、互动(收藏/点赞/评论)、下载审计、参考关联、（新）分类标签、检索(RAG) | files, file_comments, file_interactions, file_download_record, file_reference_links, file_process_tasks |
| `stats` | 周报/文件/考勤/下载统计与导出 | 跨表只读聚合 |
| `notifications` | 站内通知（横切，多域触发） | notifications |

- **不设独立 `admin` 模块**：管理后台是「按 admin 角色授权的页面」，能力由 `org`/`projects` 等域 service 提供，`/admin/*` 仅组合 + 角色门禁。
- OA 同步等外部对接，后续新增 `integrations` 模块。

### 权限（authorization）的归属

权限是**横切关注点**，分三层放置，不是简单塞进某一个模块：

| 层 | 放在哪 | 内容 |
|---|---|---|
| 鉴权原语/框架 | **`core/auth`** | 角色分级、`can(user, action, resource)`、`requireUser`/断言；统一入口 |
| 角色/组别的分配与管理 | **`org`** | 给用户分配系统角色、部门、组别的数据维护与管理 UI |
| 具体业务规则 | **各域 service** | 需本域数据的规则，如「PM 审批本项目周报」「仅上传者可改文件」「dept_ld 可见本部门项目」 |

> 即：**框架在 core/auth，管理在 org，规则在各域**。原因：资源级规则离不开本域数据，无法全部集中。
```
