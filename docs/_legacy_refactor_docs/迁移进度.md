## 0. 工作原则

- **本期策略**：先**忠实迁移**旧版全部功能/内容/样式到新架构（不改业务、不加新功能），全部迁完后再统一新增/重构。
- **`notifications` 模块本期跳过**：旧代码中周报审批等处对通知的调用，迁移时先省略/留 TODO，不阻塞主流程。
- **逐域推进**：一次一个域，跑通后再下一个。顺序：`auth → org → projects → weekly → files → stats`。
- **每域产物**：`schema(zod) + repo(drizzle) + service(业务+鉴权) + 传输层 + 组件/页面`。
- **遇阻即跳过并登记**：凡是「逻辑不清晰 / 缺外部配置 / 需业务拍板」的，记到第 4 节「问题登记」，先用桩(stub)或 TODO 占位，继续往下。
- 每完成一块，更新第 3 节进度与 git 提交。
