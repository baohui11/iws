## 2. 背景与目标

**背景**（示例）：业务侧需要将周报从线下表格迁入统一系统，并与项目、文件权限打通。

**目标**（示例）：

- 约束周报数据落库结构与审批状态机，便于统计与审计。
- 服务端为唯一写入入口，前端通过 Server Actions 调用。

**非目标**（示例）：本期不包含移动端离线填写、不与外部 OA 双向同步。

---

## 3. 总体架构（节选）

```text
浏览器 (Next.js App Router)
        │
        ▼
Server Actions / Route Handlers
        │
        ├── Supabase（Postgres + RLS + Auth）
        └── 异步任务 / 检索服务（按需，本期可占位）
```

**技术选型说明（示例）**：Web 层采用 Next.js；数据层使用 Supabase 托管 Postgres；权限在数据库层用 RLS 兜底，业务规则在 Action 层校验。

---

## 4. 核心数据流（节选）

以「成员提交周报」为例（示例）：

1. Action 校验登录用户与项目成员身份。
2. 写入 `weekly_reports`（或等价表）及子表工作项；状态置为 `pending_review`。
3. 通过 Realtime 或轮询通知 PM（实现方式待选型）。

**状态机（示例）**：`draft` → `pending_review` → `approved` / `rejected` →（可编辑后）再次 `pending_review`。

---

## 5. 接口与集成（节选）

| 类型 | 说明 |
|------|------|
| 对内 API | Server Actions 封装，不对外暴露裸 SQL。 |
| Auth | Supabase Auth JWT；`public.users` 与 `auth.users` 通过 `auth_id` 关联。 |
| 文件 | 走 Storage 签名 URL；单文件大小受 Storage 与业务配置双重限制。 |

---

## 6. 安全与非功能（节选）

- **权限**：按项目成员角色校验；敏感操作走 `service_role` 仅限服务端。
- **审计**：关键状态变更建议记录操作者与时间（表或日志方案在详细设计中展开）。
- **性能**：列表分页；大报表走异步或快照（与 PRD 统计章节对齐）。

---

## 7. 风险与待决事项（示例）

| 风险 | 缓解思路 |
|------|----------|
| 并发编辑同一周报 | 乐观锁或「仅草稿可多人协作」策略 |
| 邀请邮件到达率 | 依赖邮件网关；提供管理员重发入口 |

**待决**：全局周报截止时间是否强约束前端按钮，或仅提示 + 后端校验。
